{
- [ ] `セッションの仕組み`とは？

👉 **ブラウザとサーバーが、同じユーザーの情報をつなげて管理する流れのこと**

**解説：**
セッションの仕組みは、ユーザーごとに発行されたセッションIDを使って、サーバー側の情報を取り出せるようにする仕組みです。
ブラウザには主にセッションIDが保存され、実際のログイン情報などはサーバー側で管理されます。
ログイン状態やカート内容を維持したいときによく使われます。

具体例：

```ruby
session[:user_id] = current_user.id
```

このコードでは、ログインしているユーザーを次のリクエストでも識別できるように、セッションに `user_id`を保存するために `session` を使っています。

---

- [ ] `ユーザーがWebサイトにアクセス`とは？

👉 **セッションの流れが始まる最初の通信のこと**

**解説：**
ユーザーが最初にWebサイトへアクセスすると、ブラウザからサーバーへリクエストが送られます。
この時点では、まだセッションが作られていない場合もあります。
ここが、セッション管理のスタート地点になります。

具体例：

```http
GET / HTTP/1.1
Host: example.com
```

このコードでは、ユーザーがWebサイトにアクセスして、サーバーへ最初のリクエストを送っていることを表しています。

---

- [ ] `新しいセッションを作成する`とは？

👉 **ユーザーごとの情報を管理するための入れ物をサーバー側で用意すること**

**解説：**
初回アクセス時などに、サーバーはそのユーザー専用のセッションを作成します。
このセッションに、あとでログイン情報や一時的な状態を保存していきます。
Railsでは `session` を通してこの仕組みを使います。

具体例：

```ruby
session[:cart_item_ids] ||= []
```

このコードでは、そのユーザー用のセッションの中に、カート情報を保存する場所を用意するために `session`を使っています。

---

- [ ] `セッションID`とは？

👉 **どのユーザーのセッションかを見分けるための識別子**

**解説：**
セッションIDは、サーバー側にあるセッション情報とブラウザを結びつけるための番号のようなものです。
サーバーはこのIDを見て、どのセッション情報を使えばよいか判断します。
セッションID自体が重要な鍵になるため、漏れると危険です。

具体例：

```http
Cookie: _app_session=abc123xyz
```

このコードでは、ブラウザが保存しているセッションIDをサーバーに送って、どのセッション情報を使うか識別できるようにしています。

---

- [ ] `セッションIDの発行`とは？

👉 **サーバーがユーザー識別用のIDを作って渡すこと**

**解説：**
セッションが作られると、サーバーはそのセッションに対応するセッションIDを発行します。
このIDは、以後のリクエストで同じユーザーとして扱うために使われます。
実際のアプリでは、このIDはCookie経由でブラウザに保存されることが多いです。

具体例：

```http
Set-Cookie: _app_session=abc123xyz; Path=/; HttpOnly
```

このコードでは、サーバーがセッションIDをブラウザに渡して保存させるために `Set-Cookie` を使っています。

---

- [ ] `Cookieを使ってセッションIDを送信する`とは？

👉 **セッションIDをブラウザに保存させるためにCookieを使うこと**

**解説：**
サーバーは、セッションIDをブラウザへ渡すときにCookieを使うことが多いです。
ブラウザはそのCookieを保存し、次回以降のリクエストで自動的に送信します。
Cookieには情報そのものではなく、セッションIDだけを入れることが多いです。

具体例：

```http
Set-Cookie: _app_session=abc123xyz
```

このコードでは、ブラウザが次回以降も同じセッションを使えるように、セッションIDをCookieとして保存させています。

---

- [ ] `ユーザー情報の保存`とは？

👉 **ログイン状態やカート内容などをサーバー側に覚えさせること**

**解説：**
セッションでは、ユーザーが操作した結果をサーバー側に保存できます。
たとえば、ログイン中のユーザーIDやカートの商品IDなどを保持できます。
これにより、ページを移動しても状態を引き継げます。

具体例：

```ruby
session[:cart_item_ids] = [1, 2, 3]
```

このコードでは、ユーザーがカートに入れた商品の情報をセッションに保存して、次のページでも使えるようにしています。

---

- [ ] `セッションIDの送信`とは？

👉 **ブラウザが保存しているセッションIDを次のリクエストで送り返すこと**

**解説：**
ブラウザは、保存しているCookieを次回のリクエストに自動で含めます。
サーバーはその中のセッションIDを見て、対応するセッション情報を取り出します。
これにより、同じユーザーとして処理を続けられます。

具体例：

```http
GET /mypage HTTP/1.1
Host: example.com
Cookie: _app_session=abc123xyz
```

このコードでは、ブラウザが保存していたセッションIDをリクエストに含めて送り、サーバーが同じユーザーとして識別できるようにしています。

---

- [ ] `対応するセッション情報を取得する`とは？

👉 **サーバーがセッションIDを手がかりに保存済みの情報を取り出すこと**

**解説：**
サーバーは送られてきたセッションIDを使って、そのユーザーのセッション情報を探します。
その情報をもとに、ログイン済みかどうかや保存済みデータを確認します。
これが、状態を維持できる理由です。

具体例：

```ruby
if session[:user_id]
  @current_user = User.find(session[:user_id])
end
```

このコードでは、セッションに保存された `user_id` を使って、現在のログインユーザーを取得しています。

---

- [ ] `セッションの終了`とは？

👉 **セッションとして保存していた情報を使わなくすること**

**解説：**
ログアウト時や一定時間操作がないときに、セッションは終了します。
セッションが終了すると、保存していたログイン状態などは破棄されます。
そのため、次回は新しいセッションとして扱われます。

具体例：

```ruby
reset_session
```

このコードでは、現在のセッション情報を破棄して、ログアウト後などに以前の状態を引き継がないようにしています。

---

- [ ] `ログアウトでセッションを破棄する`とは？

👉 **ログイン状態を終わらせるために、セッション情報を消すこと**

**解説：**
ログアウト時は、以前のセッション情報が残らないようにする必要があります。
そのため、セッションを削除したりリセットしたりします。
これをしないと、意図せずログイン状態が続く可能性があります。

具体例：

```ruby
session.delete(:user_id)
```

このコードでは、ログイン状態を表す `user_id` をセッションから削除して、ログアウト状態にするために `session.delete` を使っています。}
