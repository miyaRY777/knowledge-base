{ - [ ] テストDBに db:seed を実行しない理由は？  👉 **テストの再現性（毎回同じ結果になること）を保つため**  **解説：** `db:seed` は開発用・本番用のデータを入れるためのもので、テスト環境に入れると最初からデータが存在する状態になります。 その結果、テストの前提が崩れたり、実行順によって結果が変わる不安定なテストになります。テストでは必要なデータだけを自分で作るのが基本です。  具体例：  ``` # 何もない状態を前提にするテスト expect(User.count).to eq(0) ```  このコードでは、データが存在しない前提でテストしているため、db:seed を実行していると期待通りに動かなくなります。 - [ ] destroyにおいて、 count より size が適切なのはなぜ？  👉 **関連データの件数確認では count より size を使う方が意図に合うことがあるということ**  **解説：** `count` は毎回DBに問い合わせますが、`size` はすでに読み込まれている場合はメモリ上のデータを使います。 `destroy` 後のテストでは、読み込み済みデータとのズレを防ぐために `size` の方が自然な結果になる場合があります。  具体例：  ``` user.posts.destroy_all  user.posts.size ```  このコードでは、削除後の関連データの件数を、読み込み済みの状態も含めて正しく確認するために size を使っています。  ---  - [ ] `Slug`とは？  👉 **URLに使うための、人間が読める識別子（文字列）**  **解説：** Slug は、IDの代わりにURLに使う文字列です。 `/posts/1` の代わりに `/posts/hello-world` のようにすることで、内容がわかりやすくなります。重複しないように一意性に注意が必要です。  具体例：  ``` post = Post.find_by(slug: "hello-world") ```  このコードでは、IDではなく slug を使ってデータを取得するために Slug を使っています。}
