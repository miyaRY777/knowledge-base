{ - [ ] `Lax`とは？  👉 **SameSite属性の値で、通常の遷移ではCookieを送るが、外部からの一部リクエストでは送らない設定**  **解説：** `Lax` は、別サイトからのリクエストでも「リンククリックなどの通常の遷移」ではCookieを送りますが、`POST`などの一部リクエストでは送られません。 CSRF対策としてバランスの良い設定で、Railsでもデフォルトとして使われることがあります。  **具体例：** - 他サイトからリンクをクリックして自分のサイトに移動 → Cookieは送られる - 他サイトから自動送信されたフォーム（POST） → Cookieは送られない  この例では、安全性と利便性を両立するために `Lax` が関係しています。  ---  - [ ] `Strict`とは？  👉 **SameSite属性の値で、完全に同一サイトからのリクエストだけにCookieを送る設定**  **解説：** `Strict` は、別サイトから来たリクエストでは一切Cookieを送らない設定です。 そのためセキュリティは高いですが、外部リンクからアクセスしたときにログイン状態が維持されないことがあります。  **具体例：** - 他サイトのリンクから自分のサイトにアクセス → Cookieは送られない - 同じサイト内の移動 → Cookieは送られる  この例では、セキュリティを最優先にするために `Strict` が関係しています。  ---  - [ ] `None`とは？  👉 **SameSite属性の値で、どのサイトからのリクエストでもCookieを送る設定**  **解説：** `None` は、クロスサイト（別サイト）からのリクエストでも常にCookieを送る設定です。 ただし、`Secure`属性（HTTPS限定）とセットで使う必要があります。  **具体例：** - 外部サイトのiframeやAPI連携 → Cookieが送られる - 他サイトからのリクエスト → Cookieが送られる  この例では、外部サービス連携などを実現するために `None` が関係しています。  ---  `CSRF`とは？  👉 **ユーザーの意図しないリクエストを勝手に送らせる攻撃**  **解説：** CSRF は、ユーザーがログインしている状態を悪用して、別サイトから不正な操作を実行させる攻撃です。 ユーザー本人は何もしていないのに、「ログイン済みの権限」で処理が実行されてしまうのが特徴です。  **具体例：** - ユーザーが銀行サイトにログインしたまま、別の悪意あるサイトを開く - そのページに「送金処理を行うフォーム」が仕込まれている - ページを開いた瞬間に、自動で送金リクエスト（POST）が送信される - サーバーは「ログイン中の本人の操作」と判断して、送金が実行されてしまう  この例では、ユーザーの意思とは関係なく不正な操作を実行させるために `CSRF` が使われています。  【Railsで防ぐイメージ】  ```erb <%= form_with model: @post do |f| %>   <%= f.text_field :title %> <% end %> ```  このコードでは、フォームに「CSRFトークン」が自動で含まれます。 サーバー側はこのトークンを検証することで、「本当に自分のサイトから送られたリクエストか」を確認し、CSRF攻撃を防いでいます。  ---  - [ ] `クロスサイト送信`とは？  👉 **別のサイトから自分のサイトに対してリクエストが送られること**  **解説：** クロスサイト送信とは、ユーザーが見ているサイトとは別のドメインから、リクエストが送られることです。 これ自体は正常な通信でも起こりますが、CSRFのような攻撃にも使われるため注意が必要です。  **具体例：** - 悪意のあるサイトにアクセスしたとき、自動で自分のサイトにリクエストが送られる - 外部サイトに埋め込まれたフォームやスクリプトが、自分のサイトにアクセスする  この例では、別サイトからのリクエストの仕組みを理解するために `クロスサイト送信` が関係しています。  `論理削除フラグ`とは？  👉 **データを本当に消さずに、「削除済み」として扱うための目印**  **解説：** `論理削除フラグ` は、Railsの特別な機能名ではありません。 DBのレコードを物理的に削除せず、`deleted` や `is_deleted` のようなカラムで「削除済みかどうか」を判断するために使います。 ただし、一覧表示や検索時に「削除済みのデータを除外する処理」を忘れると、消したはずのデータが表示されることがあります。  ```ruby class User < ApplicationRecord   scope :active, -> { where(deleted: false) } end ```  このコードでは、削除済みではないユーザーだけを取得するために `論理削除フラグ` を使っています。  参考：Railsの `destroy` は破棄時コールバックを実行しますが、`delete` では破棄時コールバックは無視されるとRailsガイドに説明されています。([Railsガイド](https://railsguides.jp/active_record_callbacks.html))  ---  - [ ] `モンキーテスト`とは？  👉 **決まった手順を作らず、ランダムに操作して不具合を探すテスト**  **解説：** `モンキーテスト` は、ユーザーが予想外の操作をしたときに、画面が壊れたりエラーになったりしないかを確認するテストです。 Railsの特別な機能名ではなく、ソフトウェアテスト全般で使われる考え方です。 仕様通りに確認するテストではなく、「雑に触っても壊れないか」を見るイメージです。  **具体例：** - フォームに空文字や長すぎる文字を入れる - ボタンを連打する - ページ遷移中に戻るボタンを押す - 想定外の順番で操作する  この例では、想定外の操作でもアプリが壊れないか確認するために `モンキーテスト` が関係しています。  参考：ISTQBの用語集では、モンキーテストは大きな入力範囲からランダムに選んで行うテストとして説明されています。([ISTQB Glossary](https://glossary.istqb.org/))  ---  - [ ] `論理削除`とは？  👉 **データベースから完全に消さず、削除済みとして扱う方法**  **解説：** `論理削除` は、レコード自体はDBに残したまま、`deleted_at` や `discarded_at` などのカラムで「削除済み」と判断する方法です。 復元したいデータや、履歴として残したいデータがあるときに使われます。 Rails標準の特別な削除機能ではないため、自分で実装するか、`discard` などのgemを使うことがあります。  ```ruby user.update(deleted_at: Time.current) ```  このコードでは、ユーザーのレコードをDBから消さずに、削除された時刻を保存するために `論理削除` を使っています。  参考：`discard` gemでは、ActiveRecord向けにレコードを隠す・復元する・検索するためのメソッドやスコープを提供すると説明されています。([discard GitHub](https://github.com/jhawthorn/discard))}
