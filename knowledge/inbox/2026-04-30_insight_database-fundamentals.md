`データベース`とは？

👉 **目的に合わせて整理されたデータの集まり**

**解説：**
`データベース` は、ユーザー情報・商品情報・投稿データなどを、あとから検索・更新しやすい形でまとめて保存するものです。
Webアプリでは、ユーザー登録情報や投稿内容などを保存する場所としてよく使われます。
Oracle Database の概念資料でも、Oracle Database はデータ管理のためのDBMSとして説明されています。 ([Oracle Docs](https://docs.oracle.com/en/database/oracle/oracle-database/21/cncpt/database-concepts.pdf "Database Concepts"))

**具体例：**
- ユーザーの名前・メールアドレスを保存する
- ECサイトの商品名・価格・在庫数を保存する
- SNSの投稿内容・コメント・いいね数を保存する

この例では、アプリで使うデータを整理して保存するために `データベース` が関係しています。

👉 **データベースを管理・操作するためのソフトウェア**

**解説：**
`DBMS` は、データベースに対して保存・検索・更新・削除などを行うための仕組みです。
代表例には MySQL、PostgreSQL、Oracle Database などがあります。
MySQLは、SQLを使ってデータベースを作成・管理するRDBMSとして説明されています。 ([Oracle](https://www.oracle.com/jp/mysql/what-is-mysql/ "MySQL: その定義と用途"))

**具体例：**
- PostgreSQLを使ってRailsアプリのデータを保存する
- MySQLを使ってユーザー情報を管理する
- Oracle Databaseを使って企業の業務データを管理する

この例では、データベースを実際に管理・操作するために `DBMS` が関係しています。

```sql
SELECT * FROM users WHERE email = 'sample@example.com';
```

- [ ] `トランザクション`とは？

👉 **複数のデータ操作を、ひとまとまりとして安全に扱う仕組み**

**解説：**
`トランザクション` は、データベースへの複数の操作を「全部成功する」か「全部失敗する」かのどちらかで扱うための仕組みです。
途中でエラーが起きた場合は `ROLLBACK` で変更を取り消し、成功した場合は `COMMIT` で変更を確定します。
PostgreSQLでは、`BEGIN` でトランザクションを開始し、`COMMIT` または `ROLLBACK` で終了すると説明されています。 ([PostgreSQL](https://www.postgresql.org/docs/current/sql-begin.html "PostgreSQL: Documentation: 18: BEGIN"))

```ruby
ActiveRecord::Base.transaction do
  user.update!(points: user.points - 100)
  order.create!(user: user, item: item)
end
```

このコードでは、ポイント減算と注文作成をまとめて成功・失敗させるために `トランザクション` を使っています。

**具体例：**
- 銀行振込で「Aさんの残高を減らす」と「Bさんの残高を増やす」をセットで扱う
- ECサイトで「在庫を減らす」と「注文データを作る」をセットで扱う
- ポイント利用で「ポイントを減らす」と「購入履歴を作る」をセットで扱う

この例では、片方だけ成功してデータが矛盾しないようにするために `トランザクション` が関係しています。

`ロールバック` が取り消しなら、`コミット` は確定のイメージです。
PostgreSQL公式ドキュメントでは、トランザクションは `COMMIT` によって終了できると説明されています。 ([PostgreSQL](https://www.postgresql.org/docs/current/tutorial-transactions.html "Documentation: 18: 3.4. Transactions"))

```sql
COMMIT;
```

このコードでは、トランザクション中の変更を確定するために `コミット` を使っています。

PostgreSQL公式ドキュメントでは、`ROLLBACK` は現在のトランザクションで行われた更新を破棄すると説明されています。 ([PostgreSQL 日本](https://www.postgresql.jp/docs/14/sql-rollback.html "ROLLBACK"))

このコードでは、トランザクション中に行った変更を取り消すために `ロールバック` を使っています。

- [ ] `データの整合性 / 一貫性`とは？

👉 **データに矛盾や不正な状態がなく、決められたルールに沿ってそろっていること**

**解説：**
`データの整合性` は、データベース内の情報が正しく、矛盾していない状態を指します。
`データの一貫性` は、システム全体でデータの状態やルールがそろっていることに注目する考え方です。
例えば、存在しないユーザーに紐づく投稿が作られないようにすることや、退会済みユーザーをすべての機能で退会済みとして扱うことも、整合性や一貫性を守る例です。
PostgreSQLの制約では、条件に違反するデータを保存しようとするとエラーになると説明されています。 ([PostgreSQL](https://www.postgresql.org/docs/current/ddl-constraints.html "Documentation: 18: 5.5. Constraints"))

**具体例：**
- 同じメールアドレスのユーザーを重複登録させない
- 存在しないユーザーIDを投稿データに入れない
- 注文データだけ作られて、支払いデータが作られない状態を防ぐ
- どの画面でも同じユーザー情報が表示される

この例では、データの矛盾を防ぎ、システム全体で同じルールを保つために `データの整合性 / 一貫性` が関係しています。

- [ ] `データの重複`とは？

👉 **同じ意味のデータが複数の場所に存在している状態**

重複が多いと、片方だけ更新されて情報がずれる原因になります。

**具体例：**
- 顧客Aの住所が、営業用シートと請求用シートの両方にある
- 商品名が複数のテーブルやファイルに別々に保存されている
- 同じユーザー情報を複数箇所で手入力している

この例では、情報のズレや更新漏れが起きやすくなるために `データの重複` が問題になります。

- [ ] `データの矛盾`とは？

👉 **同じはずの情報が場所によって違っている状態**

**解説：**
`データの矛盾` は、本来一致しているべきデータが、更新漏れなどによって食い違っている状態です。

この例では、同じ情報が食い違っているために `データの矛盾` が起きています。

```ruby
add_column :users, :name, :string, null: false
```

このコードでは、`users` テーブルの `name` に空の値を保存できないようにするために `NOT NULL制約` を使っています。

- [ ] `ロック機構`とは？

👉 **同時に同じデータを変更して壊れないようにする仕組み**

**解説：**
`ロック機構` は、複数のユーザーや処理が同時に同じデータを操作するときに、データの競合を防ぐための仕組みです。
PostgreSQLでは、テーブル内のデータへの同時アクセスを制御するために、複数のロックモードが用意されています。 ([PostgreSQL](https://www.postgresql.org/docs/current/explicit-locking.html "Documentation: 18: 13.3. Explicit Locking"))

**具体例：**
- Aさんが在庫数を更新している間、Bさんの更新を待たせる
- 同じ銀行口座の残高を同時に変更しないようにする
- チケット残数を複数人が同時購入しても、数がずれないようにする

この例では、同時操作によるデータの不整合を防ぐために `ロック機構` が関係しています。

- [ ] `権限管理 / アクセス制御`とは？

👉 **誰がどのデータを操作できるかを決める仕組み**

**解説：**
`権限管理` や `アクセス制御` は、ユーザーや役割ごとに、データの閲覧・追加・更新・削除などを制限する仕組みです。
PostgreSQLでは、ロールを使ってデータベースへのアクセス権限を管理し、`GRANT` で権限を付与できます。 ([PostgreSQL](https://www.postgresql.org/docs/current/user-manag.html "Documentation: 18: Chapter 21. Database Roles"))

```sql
GRANT SELECT ON users TO readonly_user;
```

このコードでは、`readonly_user` に `users` テーブルを閲覧する権限を与えるために `権限管理 / アクセス制御` を使っています。

**具体例：**
- 管理者だけがユーザー一覧を見られる
- 一般ユーザーは自分の投稿だけ編集できる
- 読み取り専用ユーザーはデータを変更できない

この例では、データを安全に扱うために `権限管理 / アクセス制御` が関係しています。

- [ ] `認証`とは？

👉 **その人が本人かどうかを確認する仕組み**

**解説：**
`認証` は、ログイン時にメールアドレスやパスワードなどで本人確認をする仕組みです。
データベースの権限管理でも、ユーザー名やパスワードなどを使って、誰がアクセスしているかを確認します。

**具体例：**
- メールアドレスとパスワードでログインする
- GitHubアカウントでログインする
- データベース接続時にユーザー名とパスワードを使う

この例では、アクセスしている人を確認するために `認証` が関係しています。

- [ ] `バックアップ・リカバリ`とは？

👉 **データを失わないように保存し、問題が起きたときに復元する仕組み**

**解説：**
`バックアップ` は、障害やミスに備えてデータのコピーを保存することです。
`リカバリ` は、サーバー障害・誤操作・データ破損などが起きたときに、バックアップなどを使って正常な状態へ戻す作業です。
PostgreSQLでは、SQLダンプ・ファイルシステムレベルのバックアップ・継続的アーカイブというバックアップ方法が説明されています。 ([PostgreSQL](https://www.postgresql.org/docs/current/backup.html "Documentation: 18: Chapter 25. Backup and Restore"))

**具体例：**
- 本番データベースを定期的に保存する
- 誤って削除したデータをバックアップから戻す
- サーバー障害が起きたときに、別環境でデータを復元する

この例では、障害や操作ミスからデータを守るために `バックアップ・リカバリ` が関係しています。

- [ ] `一元管理`とは？

👉 **データをバラバラにせず、1つの場所でまとめて管理すること**

**解説：**
`一元管理` は、同じ情報を複数の場所に重複して持たず、中心となる場所で管理する考え方です。
データベースを使うと、複数の画面やアプリから同じデータを参照しやすくなります。

**具体例：**
- 顧客情報をExcelごとに分けず、1つのデータベースで管理する
- プロフィール情報を複数画面で同じデータから表示する
- 商品情報を管理画面と購入画面で共通利用する

この例では、データの重複や矛盾を減らすために `一元管理` が関係しています。

- [ ] `リアルタイム更新`とは？

👉 **変更されたデータがすぐに他の利用者や画面にも反映されること**

**解説：**
`リアルタイム更新` は、誰かがデータを変更したときに、他のユーザーやアプリケーションでもすぐに最新状態を使えることです。
データベースを共有していると、同じ元データを参照できるため、情報のズレを減らしやすくなります。

**具体例：**
- 管理者が商品価格を変更すると、購入画面にも新しい価格が表示される
- サポート担当が顧客情報を更新すると、営業担当も最新情報を見られる
- 在庫数が減ると、販売画面にも残り数が反映される

この例では、最新のデータを共有するために `リアルタイム更新` が関係しています。

- [ ] `表計算ソフト`とは？

👉 **表形式でデータを入力・計算・管理できるソフト**

**解説：**
`表計算ソフト` は、ExcelやGoogleスプレッドシートのように、行と列でデータを扱うソフトです。
少量のデータを人が直接編集するには便利ですが、複数人・複数アプリで厳密にデータを管理する場合は、重複や矛盾が起きやすくなります。

**具体例：**
- Excelで顧客一覧を管理する
- Googleスプレッドシートで売上表を作る
- 手作業で住所や電話番号を更新する

この例では、手軽に表形式のデータを扱うために `表計算ソフト` が関係しています。
